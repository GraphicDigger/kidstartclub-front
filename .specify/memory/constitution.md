# Конституция KidStartClub Frontend

## Основные принципы

### I. Feature-Sliced Design (ОБЯЗАТЕЛЬНО)

Структура проекта:

```
src/
├── app/          # Next.js App Router: страницы, layouts, стор
├── entities/     # Доменные сущности: api/, model/, ui/, types.ts, index.ts
├── features/     # Фичи
├── widgets/      # Составные компоненты страниц (Header, Hero, PostGrid…)
├── games/        # Игровые модули (dotByDot с Redux)
└── shared/
    ├── uiKit/    # UI-кит: Button, Card, Dialog, Stack, Grid, Typography, Scroll…
    ├── theme/    # Дизайн-токены: sys.js (spacing, colors, breakpoints), comp-*.js
    ├── providers/# EmotionProvider, ThemeProvider, StoreProvider
    ├── icons/
    ├── lib/      # Хуки и утилиты
    └── animation/
```

Весь код ДОЛЖЕН соблюдать границы слоёв FSD: `app` → `widgets` → `features` → `entities` → `shared`.
Верхние слои МОГУТ импортировать из нижних; нижние слои НЕ ДОЛЖНЫ импортировать из верхних.
Каждый слайс ОБЯЗАН открывать публичный API исключительно через `index.ts`/`index.js` —
прямые импорты внутренних путей слайса извне запрещены.
Новые доменные сущности ДОЛЖНЫ создаваться через `npm run gen` (Plop), не вручную.

**Обоснование**: FSD закрепляет явное владение кодом и предотвращает сквозные зависимости,
которые делают рефакторинг больших React-приложений дорогостоящим.

### II. Emotion как основа стилизации

Все стили компонентов ДОЛЖНЫ использовать Emotion `styled()`. SCSS-модули допустимы
только для layout-утилит (например, Stack). Все цвета, отступы и брейкпоинты ОБЯЗАНЫ
использовать токены темы (`theme.sys`, `theme.comp`) — хардкод значений запрещён.
Динамические стили ДОЛЖНЫ использовать transient-пропсы (`$propName`), чтобы избежать
передачи пропсов в DOM. Логика вариантов ДОЛЖНА быть оформлена единым блоком
`${({ $prop }) => $prop && `...`}` на каждый вариант, а не тернарным выражением на каждое свойство.

**Обоснование**: Стили находятся рядом с компонентом, интерфейс остаётся
согласованным с темой, исключаются конфликты специфичности при смешении систем стилизации.

### III. Строгий TypeScript

Все новые файлы ДОЛЖНЫ быть TypeScript (`.ts`/`.tsx`). Использование `any` запрещено
без комментария `// eslint-disable` с явным обоснованием. Пропсы компонентов ОБЯЗАНЫ
быть типизированы через интерфейсы или псевдонимы типов. `npm run typecheck` ДОЛЖЕН
проходить без ошибок до того, как фича считается завершённой.

**Обоснование**: Типобезопасность предотвращает ошибки в рантайме и делает API
компонентов самодокументирующимся без необходимости в prose-комментариях.

### IV. Дисциплина API компонентов

Все компоненты `uiKit` ДОЛЖНЫ использовать `React.forwardRef`. Каждый компонент или
директория слайса ОБЯЗАНЫ открывать публичный API через `index.ts`/`index.js` —
глубокие импорты извне запрещены. Для каждого нового компонента `uiKit` ДОЛЖНА быть
создана Storybook-история (`Component.stories.js`) рядом с файлом компонента.

**Обоснование**: `forwardRef` обеспечивает компонуемость; экспорты через index позволяют
рефакторить внутренности без нарушения потребителей; stories служат живой документацией.

### V. Доступность и анимации

Интерактивные компоненты ДОЛЖНЫ использовать примитивы Radix UI для корректной семантики,
навигации с клавиатуры и управления фокусом. Анимации ДОЛЖНЫ использовать GSAP;
CSS-переходы допустимы только для микроинтеракций (hover-состояния, смена opacity).
Drag-and-drop ДОЛЖЕН использовать `@hello-pangea/dnd`. Добавление новых библиотек
анимаций или DnD запрещено.

**Обоснование**: Radix корректно обрабатывает ARIA-атрибуты и захват фокуса по умолчанию;
GSAP и @hello-pangea/dnd покрывают все сложные интерактивные потребности проекта.

## Интеграция с бэкендом

**Репозиторий бэкенда**: `/Users/ilmirsultanov/Projects/kidstartclub/back`
**Стек**: Django 4.2 + Django Ninja ≥ 1.3.0, Python 3.11
**База данных**: SQLite (разработка) / MySQL (продакшн)

### Адреса API

| Среда | URL |
|-------|-----|
| Разработка | `http://127.0.0.1:8000` |
| Продакшн | `https://api.kidstart.club` |

Базовый URL задаётся через переменную окружения `DJANGO_API_URL` (серверная, без префикса
`NEXT_PUBLIC_`). Дефолт: `http://127.0.0.1:8000`.

### Доступные эндпоинты

**Курсы** (префикс `/api/courses`):

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/courses` | Список курсов; query: `category`, `subcategory`, `page` |
| GET | `/api/courses/{slug}` | Детали курса |
| GET | `/api/courses/categories` | Все категории |
| GET | `/api/courses/subcategories` | Все подкатегории |

**Посты** (префикс `/api/posts`):

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/posts` | Список постов; query: `category`, `page` |
| GET | `/api/posts/{slug}` | Детали поста |
| GET | `/api/posts/categories` | Все категории |

Пагинированные эндпоинты возвращают `{ items: [...], count: N }` (Django Ninja `PageNumberPagination`,
`page_size=12`). Непагинированные возвращают массив напрямую.

### Соглашения по работе с API на фронте

- Все вызовы API ДОЛЖНЫ располагаться в `src/entities/<entity>/api/` соответствующего слайса FSD.
- Интерфейс `ApiXxx` ОБЯЗАН точно отражать схему Django Ninja — не допускать расхождений.
- Маппинг из `ApiXxx` в доменный тип ДОЛЖНА выполнять функция `mapApiXxx()` внутри того же файла.
- Медиа-пути ДОЛЖНЫ нормализоваться через хелпер `mediaUrl()`: относительный путь → `${API_URL}${path}`.
- Запросы к бэкенду ДОЛЖНЫ использовать Next.js `fetch` с `{ next: { revalidate: 60 } }` для ISR,
  если данные не требуют реального времени.
- Аутентификация на публичных эндпоинтах отсутствует; при появлении защищённых роутов —
  схема аутентификации фиксируется в этом разделе.

### Критерии качества для API-слоя

Следующие условия блокируют мёрж:

- Прямые `fetch`-вызовы к бэкенду за пределами `entities/<entity>/api/`
- Хардкодированный URL бэкенда (`http://127.0.0.1:8000` или `api.kidstart.club`) вне `.env`-переменной
- Отсутствие интерфейса `ApiXxx` для нового эндпоинта (типизация ответа `any` или `unknown` без приведения)
- Медиа-URL без нормализации через `mediaUrl()` или аналог

## Процесс разработки

Каждый вклад ДОЛЖЕН пройти следующие проверки перед мёржем:

- `npm run lint` — ноль ESLint-ошибок
- `npm run typecheck` — ноль TypeScript-ошибок
- `npm run build` — продакшн-сборка без ошибок
- Визуальные изменения ДОЛЖНЫ быть проверены в Storybook (`npm run storybook`) до апрува PR
- Новые слайсы `entities/` ДОЛЖНЫ создаваться через `npm run gen`, не вручную

Intercepting Routes (паттерн модалка + standalone URL) ОБЯЗАНЫ включать
`useEffect(() => router.refresh(), [])` при монтировании — см. `src/app/book-club/README.md`.

## Критерии качества

Следующие условия блокируют мёрж любого pull request:

- Любое нарушение границ слоёв FSD (нижний слой импортирует верхний)
- Любое хардкодированное значение цвета, отступа или брейкпоинта вне системы темы
- Любой компонент `uiKit` без `React.forwardRef`
- Ошибка `typecheck` или `build`
- Новый слайс без файла публичного API `index.ts`/`index.js`

## Управление

Данная конституция имеет приоритет над всеми прочими соглашениями, задокументированными в репозитории.

Для внесения поправок требуется:
1. Pull request с изменением `.specify/memory/constitution.md`
2. Чёткое техническое обоснование (не стилистическое предпочтение)
3. Примечание о миграции для существующего кода, нарушающего новое правило

**Версионирование** (семантическое):
- MAJOR: Удаление или несовместимое переопределение существующего принципа
- MINOR: Добавление нового принципа или существенное расширение руководства
- PATCH: Уточнение формулировки или форматирование

Соответствие конституции проверяется в ходе ревью PR. Авторы ОБЯЗАНЫ
самостоятельно проверить соответствие перед запросом ревью.
