# Задачи: [FEATURE NAME]

**Входные данные**: Проектные документы из `/specs/[###-feature-name]/`
**Предварительные условия**: plan.md (обязательно), spec.md (обязательно для user story), research.md, data-model.md, contracts/

**Тесты**: Примеры ниже включают тестовые задачи. Тесты ОПЦИОНАЛЬНЫ — включайте их только если это явно указано в спецификации фичи.

**Организация**: Задачи сгруппированы по user story для независимой реализации и тестирования каждой истории.

## Формат: `[ID] [P?] [Story] Описание`

- **[P]**: Можно выполнять параллельно (разные файлы, нет зависимостей)
- **[Story]**: К какой user story относится задача (например, US1, US2, US3)
- Указывайте точные пути к файлам в описаниях

## Соглашения по путям

- **Один проект**: `src/`, `tests/` в корне репозитория
- **Веб-приложение**: `backend/src/`, `frontend/src/`
- **Мобильное**: `api/src/`, `ios/src/` или `android/src/`
- Пути ниже рассчитаны на один проект — корректируйте по структуре из plan.md

## Зависимости и порядок выполнения

### Зависимости между фазами

- **Настройка (Фаза 1)**: Нет зависимостей — можно начинать сразу
- **Фундамент (Фаза 2)**: Зависит от завершения Настройки — БЛОКИРУЕТ все user story
- **User Stories (Фаза 3+)**: Все зависят от завершения Фундамента
  - User story могут выполняться параллельно (при наличии ресурсов)
  - Или последовательно в порядке приоритета (P1 → P2 → P3)
- **Полировка (Финальная фаза)**: Зависит от завершения всех нужных user story

### Зависимости между user story

- **User Story 1 (P1)**: Можно начать после Фундамента — нет зависимостей от других историй
- **User Story 2 (P2)**: Можно начать после Фундамента — может интегрироваться с US1, но должна тестироваться независимо
- **User Story 3 (P3)**: Можно начать после Фундамента — может интегрироваться с US1/US2, но должна тестироваться независимо

### Внутри каждой user story

- Тесты (если включены) ДОЛЖНЫ быть написаны и ПАДАТЬ до реализации
- Модели — до сервисов
- Сервисы — до эндпоинтов
- Основная реализация — до интеграции
- История завершена — до перехода к следующему приоритету

### Возможности для параллелизма

- Все задачи Настройки с [P] можно выполнять параллельно
- Все задачи Фундамента с [P] можно выполнять параллельно (в рамках Фазы 2)
- После завершения Фундамента все user story могут начинаться параллельно (при наличии команды)
- Все тесты для user story с [P] можно запускать параллельно
- Модели внутри истории с [P] можно создавать параллельно
- Разные user story могут выполняться параллельно разными участниками команды

## Стратегия реализации

### Сначала MVP (только User Story 1)

1. Завершить Фазу 1: Настройка
2. Завершить Фазу 2: Фундамент (КРИТИЧНО — блокирует все истории)
3. Завершить Фазу 3: User Story 1
4. **ОСТАНОВИТЬСЯ И ПРОВЕРИТЬ**: Протестировать User Story 1 независимо
5. Развернуть/продемонстрировать при готовности

### Инкрементальная поставка

1. Настройка + Фундамент → Фундамент готов
2. Добавить User Story 1 → Протестировать независимо → Развернуть/Продемонстрировать (MVP!)
3. Добавить User Story 2 → Протестировать независимо → Развернуть/Продемонстрировать
4. Добавить User Story 3 → Протестировать независимо → Развернуть/Продемонстрировать
5. Каждая история добавляет ценность без поломки предыдущих

### Стратегия параллельной команды

При нескольких разработчиках:

1. Команда вместе завершает Настройку + Фундамент
2. После завершения Фундамента:
   - Разработчик A: User Story 1
   - Разработчик B: User Story 2
   - Разработчик C: User Story 3
3. Истории завершаются и интегрируются независимо

## Примечания

- Задачи [P] = разные файлы, нет зависимостей
- Метка [Story] привязывает задачу к конкретной user story для трассируемости
- Каждая user story должна быть независимо завершаемой и тестируемой
- Убедитесь, что тесты падают до начала реализации
- Коммитьте после каждой задачи или логической группы
- Останавливайтесь на каждой контрольной точке для независимой валидации истории
- Избегайте: расплывчатых задач, конфликтов по одному файлу, межисторийных зависимостей, нарушающих независимость
